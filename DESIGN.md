# DESIGN — 現状の UI / UX 設計

概要
- InstantTest は教師（作成者）と生徒（受験者）の二つの主要ペルソナを想定した軽量なテスト作成／受験アプリです。現在の実装は最小限のインターフェースで「作成→配布→受験→確認」がスムーズに行えることを優先しています。

ペルソナ
- 教師: クラス管理、テスト作成、問題編集、AI による問題生成、受験結果の確認を主なタスクとします。
- 生徒: クラス参加（コードまたは公開URL）→ 試験開始 → 回答 → 結果確認 の流れを最短で体験できることを優先します。

主要フロー
- 教師: ダッシュボード → クラス作成 → テスト作成（名前/説明/公開/ランダム化）→ 問題追加（手動/AI生成）→ 編集 → 共有（URL/QR）
- 生徒: 共有URL または クラス+コードで参加 → 試験開始（`exam_sessions` 作成）→ 回答 → 提出 → 集計／レポート閲覧

スクリーンとコンポーネント（現状）
- ダッシュボード: クラス一覧、テスト一覧、短時間で作成できる UI
- テスト編集: 問題一覧、問題作成フォーム、選択肢管理、AI 生成モーダル（生成→編集→保存）
- 受験ビュー: 一覧形式で問題を表示、進捗表示、送信ボタン
- レポート画面: 個別スコア、問題別の正誤、試行履歴
- 共有/QR モーダル: `/api/qr-code` を使った QR 表示と URL コピー

アクセシビリティとテーマ
- 高コントラストやサンプルテーマはリポジトリの `public/` に配置されています（例: `teacher_theme_high_contrast.css`）。
- キーボード操作・フォーカス管理を意識し、モーダル表示時はフォーカスを移動する実装になっています。

レスポンシブ
- 教師画面はデスクトップでの一覧性を重視し 2 カラム実装を主に想定、スマホでは 1 カラム構成に落とします。

インタラクションの要点（現在の実装）
- 削除操作: 依存データがある場合はクライアントから `?cascade=1` を付けて明示的に実行する設計（サーバー側で確認）
- ランダム化: テストの `randomize` フラグが有効ならサーバーが問題順・選択肢順をシャッフルして返します
- 試行管理: `exam_sessions` と `student_answers.session_id` によって受験を紐づけ、個別集計が可能です
- AI生成: 教師権限で `/api/generate-questions` を呼び出すワークフロー（生成は `server/geminiAi.js` を使用、`GEMINI_API_KEY` 必須）

現状の制約
- フロントエンドは最小実装（直接 DOM 操作を利用する箇所あり）。コンポーネント化やスタイル体系の整理（CSS モジュール等）は将来的な改善点です。
