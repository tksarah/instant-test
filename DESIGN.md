# DESIGN — 現状の UI / UX 設計

概要
- InstantTest は教師（作成者）と生徒（受験者）の二つの主要ペルソナを想定した軽量なテスト作成／受験アプリです。現在の実装は最小限の UI で「作成→配布→受験→確認」を速やかに行えることを重視しています。

ペルソナ
- 教師: クラス管理、テスト作成、問題編集、AI による問題生成、受験結果の確認を主なタスクとします。
- 生徒: 共有 URL またはクラスコードで参加し、素早く受験を完了できることを優先します。

主要フロー
- 教師: ダッシュボード → クラス作成 → テスト作成（名前/説明/公開/ランダム化）→ 問題追加（手動/AI生成）→ 編集 → 共有（URL/QR）
- 生徒: 共有 URL または クラス+コードで参加 → 試験開始（`exam_sessions` 作成）→ 回答 → 提出 → 集計／レポート閲覧

ページと主要 API のマッピング（現状）
- `server/public/app.html` — 教師ダッシュボード / テスト編集: `/api/teacher/me`, `/api/classes`, `/api/tests`, `/api/tests/:id/questions`, `/api/generate-questions`
- `server/public/create_test.html` — テスト作成/編集 UI: `/api/tests`, `/api/tests/:id/questions`
- `server/public/index.html` — 生徒向け公開ビュー（クラス一覧 / 参加）: `/api/classes`, `/api/students`
- `server/public/login.html` — 教師ログイン: `/api/teacher/login`, `/api/teacher/logout`

スクリーンとコンポーネント（現状）
- ダッシュボード: クラス一覧、テスト一覧、短時間で作成できる UI
- テスト編集: 問題一覧、問題作成フォーム、選択肢管理、AI 生成モーダル（生成→編集→保存）
- 受験ビュー: 問題リスト、進捗表示、送信
- レポート画面: 個別スコア、問題別正誤、試行履歴
- 共有/QR モーダル: `/api/qr-code` を使った QR 表示と URL コピー

アクセシビリティとテーマ
- 高コントラストやサンプルテーマはリポジトリの `public/` に配置されています（例: `teacher_theme_high_contrast.css`）。
- キーボード操作・フォーカス管理を考慮した設計。ただし一部 DOM 操作が直書きになっており、アクセシビリティの改善余地があります。

レスポンシブ
- 教師画面はデスクトップでの一覧性を重視し 2 カラム、モバイルでは 1 カラムへフォールバックします。

インタラクションと設計上の決定事項
- 削除操作: 依存データがある場合はクライアントが `?cascade=1` を付けて明示実行する設計（サーバーで整合性チェックを行う）
- ランダム化: テストの `randomize` フラグが有効ならサーバー側で問題と選択肢をシャッフルして返却する
- セッション管理: `exam_sessions` と `student_answers.session_id` によって受験を紐づける
- AI 生成: 教師が `/api/generate-questions` を呼び出すワークフロー（`server/geminiAi.js` を使用。`GEMINI_API_KEY` 必須）

現状の制約と今後の改善案
- フロントエンドは最小実装で、コンポーネント化（React 等の導入）やスタイル体系の整理が必要
- 入力検証や権限周りの強化（細かなロール、監査ログ）を優先で改善する
